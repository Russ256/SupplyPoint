Summary

I believe the most important part of any system is the business logic in the server api. This in my opionion is the backbone of any software and is crucial for an enterprise system to have a good server architecture providing isolation of the business logic into a single layer. If the server architecture is good then the UI (although also still a very important part of the system) can be easily changed. So the back end is where I have focused the majority of my effort for this demo. The demo consists of an Angular front end running on an ASP.NET Core web site and a back end application built using a CQRS/DDD architecture. It provides for user registration, login and authenitcation and there is a products page that displays a list of products from the database with the ability to create new products. This shows retrieval of data through the query layer and updates through the command layer. All data is held in an sqlite database. You can use the register link at the top of the page to create a user and login.

Front End

The website project contains the angular app and the controllers for calling the application services. I would normally separate the angular UI from the application api but have included them the same project for ease of developing this demo. Dependency Injection is used for all application services.

Back End

The back end is built using a CQRS architecture for api request handling and Domain Driven Design for the business logic:
  • Queries are handled using Dapper to provider high performance responses.
  • MedaitR is used to decouple the api requests from the command implementations.
  • FluentValidation is used for command request validation classes.
  • The business logic is built using DDD methodogy:
    • Entity framework is used for data updates
    • Data access is abstracted using the repository pattern
    • Transaction integrity with commit/rollback logic is implemented using the Unit Of Work pattern implemented in the MediatR request pipeline.
  • There are a few unit test to show the use of the Moq library for mocking class dependencies.
